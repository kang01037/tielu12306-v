# 铁路12306售票系统设计文档

## 1. 项目概述

### 1.1 项目背景
本项目为**纯后端RESTful API服务**，采用前后端分离架构，前端Vue3为独立项目。系统模拟中国铁路12306互联网售票系统，实现火车票查询、预订、支付、退改签等核心功能的RESTful API。重点解决高并发场景下的库存扣减、订单一致性等关键技术难题。

### 1.2 项目目标
- 提供完整的火车票在线售票业务RESTful API
- 支持高并发查票与下单，保障数据一致性
- 提供用户管理、订单管理、列车管理等后台功能接口

### 4.2 库存表设计说明

库存表 `train_seat_inventory` 是本系统最关键的表，采用**号段库存模型**：
- 每张表记录一个**车次 × 日期 × 席位类型**的库存
- 使用 `version` 乐观锁字段防止超卖
- 库存扣减使用行级锁确保原子性：
  ```sql
  UPDATE train_seat_inventory 
  SET sold_count = sold_count + #{count}, version = version + 1
  WHERE train_id = #{trainId} 
    AND travel_date = #{travelDate}
    AND seat_type = #{seatType}
    AND sold_count + #{count} <= total_count
    AND version = #{version}
  ```

## 5. 核心业务流程

### 5.1 查票流程

```
用户输入 出发站/到达站/日期
        │
        ▼
┌───────────────────┐
│  网关层限流校验    │
└───────┬───────────┘
        │
        ▼
┌───────────────────┐
│  Redis 缓存查询    │───── 命中 ────▶ 直接返回结果
│  (车次列表)        │
└───────┬───────────┘
        │ 未命中
        ▼
┌───────────────────┐
│  MySQL 查询车次    │
│  及余票信息        │
└───────┬───────────┘
        │
        ▼
┌───────────────────┐
│  计算各席位票价    │
│  更新Redis缓存     │
└───────┬───────────┘
        │
        ▼
      返回结果
```

优化策略：
- 车次基础数据缓存到 Redis，定时刷新(5分钟)
- 余票数量从 Redis 读取，实时更新
- 车次查询使用 Elasticsearch（后期优化）

### 5.2 下单流程

```
用户提交订单 (车次/日期/乘车人/席位)
        │
        ▼
┌───────────────────┐
│  参数校验          │ ── 失败 ──▶ 返回错误
│  (乘车人/车次状态)  │
└───────┬───────────┘
        │ 通过
        ▼
┌───────────────────┐
│  Redis 分布式锁    │ ── 获取失败 ──▶ 排队重试/返回繁忙
│  (车次+日期+席位)  │
└───────┬───────────┘
        │ 获取锁
        ▼
┌───────────────────┐
│  扣减库存          │ ── 库存不足 ──▶ 释放锁, 返回无票
│  (乐观锁 UPDATE)    │
└───────┬───────────┘
        │ 扣减成功
        ▼
┌───────────────────┐
│  生成订单          │
│  (状态: 待支付)     │
└───────┬───────────┘
        │
        ▼
┌───────────────────┐
│  发送延时消息       │ ── 30分钟后检查支付状态
│  (RabbitMQ)        │     未支付则自动取消
└───────┬───────────┘
        │
        ▼
    返回订单号
```

关键设计：
- 下单使用 Redis 分布式锁（车次+日期+席位粒度）避免并发冲突
- 库存扣减采用乐观锁 + 数据库唯一约束双重保障
- 订单创建后发送 RabbitMQ 延迟消息，30分钟后检查是否支付，未支付自动取消并释放库存

### 5.3 支付回调流程

```
支付平台回调
        │

┌───────────────────┐
│  签名验证          │ ── 失败 ──▶ 拒绝

        │
        ▼
┌───────────────────┐

```
用户申请退票

        ▼

## 6. 高并发方案

### 6.1 查票高并发

| 策略 | 方案 |
|------|------|
| 多级缓存 | 本地Caffeine → Redis → MySQL |
| 读写分离 | MySQL主从复制，读从库 |
| CDN | 静态资源/CDN缓存车次时刻表 |
| 限流 | 网关层令牌桶限流，每用户每秒N次 |

### 6.2 下单高并发

| 策略 | 方案 |
|------|------|
| 分布式锁 | Redis Redisson锁，车次+日期+席位粒度 |
| 乐观锁 | 库存表version字段，防止超卖 |
| 库存预热 | 每日凌晨将当天库存加载到Redis |
| 异步化 | RabbitMQ异步处理订单后置逻辑 |
| 排队机制 | 高峰期超出处理能力时进入排队队列 |

### 6.3 数据一致性保证

- **库存一致性**: Redis 预扣 + MySQL 乐观锁 → 定时对账修正
- **订单一致性**: 本地事务 + RabbitMQ 事务消息
- **支付一致性**: 幂等设计 + 定时轮询 + 对账文件

## 8. 安全设计

### 8.1 认证鉴权

- **JWT Token**: 用户登录成功后返回JWT Token，有效期24小时
- **密码加密**: 使用BCrypt算法对用户密码进行加密存储
- **网关鉴权**: 网关层统一校验JWT Token，非法请求直接拦截
- **接口权限**: 敏感接口(订单、支付)需要用户登录态

### 8.2 数据安全

- **敏感信息脱敏**: 身份证号、手机号在响应中脱敏展示
- **SQL注入防护**: 使用MyBatis-Plus参数化查询
- **XSS防护**: 输入参数校验与过滤
